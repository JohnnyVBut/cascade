# Cascade — Описание возможностей

Cascade — это самостоятельно размещаемая платформа управления маршрутизатором WireGuard / AmneziaWG.
Написана с нуля на Go + Fiber, предоставляет корпоративные возможности маршрутизации, файрвола
и мониторинга через удобный веб-интерфейс.

---

## Туннельные интерфейсы

Несколько независимых интерфейсов WireGuard или AmneziaWG, каждый со своей подсетью,
портом и списком пиров.

- Создание / редактирование / удаление интерфейсов (`wg10`, `wg11`, …)
- Запуск / остановка / перезапуск через UI или API
- Два протокола: **WireGuard 1.0** (`wg-quick`) и **AmneziaWG 2.0** (`awg-quick`)
- Горячая перезагрузка параметров без разрыва активных соединений (`awg syncconf`)
- Автозапуск при перезапуске контейнера — все интерфейсы с `enabled=true` поднимаются автоматически
- Параметры обфускации AWG2 на интерфейс (Jc, Jmin, Jmax, S1–S4, H1–H4, I1–I5)
- Экспорт параметров интерфейса для S2S-сценария
- Бэкап / восстановление интерфейса со всеми пирами в JSON

---

## Пиры (Peers)

Два типа пиров на каждом интерфейсе:

### Клиентские пиры
- Автогенерация ключевой пары (приватный ключ хранится на сервере для QR/скачивания)
- Скачивание клиентского конфига (`.conf`) и QR-код
- Индивидуальный `AllowedIPs` для split-tunnel или полного туннеля
- Включение / выключение без удаления
- Опциональная дата истечения срока действия
- Одноразовая ссылка на конфиг

### Interconnect-пиры (S2S)
- Рабочий процесс Экспорт → Импорт JSON для настройки site-to-site туннелей
- PSK генерируется автоматически при импорте и включается в реэкспорт для второй стороны
- `AllowedIPs = <remote_ip>/32` для точного крипто-роутинга в топологиях с несколькими пирами
- Бейдж S2S и runtime endpoint (IP:port из `wg dump`, обновляется каждую секунду)

## Обфускация AmneziaWG 2.0

- Параметры обфускации хранятся в БД на каждый интерфейс
- **7 CPS-профилей** для имитации трафика: QUIC Initial, QUIC 0-RTT, TLS 1.3,
  DTLS 1.3, HTTP/3, SIP, Noise_IK
- Уровни интенсивности: `low`, `medium`, `high`
- **AWG2 Шаблоны** — сохранение, загрузка и передача профилей обфускации
- **Генерация (⚡)** — генерация параметров в один клик по алгоритму AmneziaWG-Architect
  с опциональным сохранением как шаблона
- Непересекающиеся диапазоны H1–H4 (4 зоны пространства uint32, без риска коллизий)

## Маршрутизация

### Состояние таблицы маршрутов ядра
- Просмотр живой таблицы маршрутизации ядра (любая таблица: `main`, `100`, `vpn_kz`, …)
- Таблицы маршрутизации обнаруживаются автоматически из `ip rule show` — ручная настройка не нужна
- Текстовый парсинг (`ip route show`), никогда `ip -j` — работает на всех версиях ядра

### Политико-зависимый поиск маршрута (Route Lookup)
- Проверка любого IP назначения с опциональным IP источника
- Симуляция PBR-правил из FirewallManager (fwmark → таблица)
- Показывает, какое правило файрвола сработало и какая таблица маршрутизации использована
- Поддержка проверки членства в ipset для алиасных правил

### Статические маршруты
- Создание постоянных маршрутов (`destination`, `via`, `dev`, `metric`, `table`)
- Включение / выключение каждого маршрута отдельно
- Переживают перезапуск контейнера — восстанавливаются после `tunnel.Init()`, когда интерфейсы уже существуют

## NAT (трансляция сетевых адресов)

### Исходящий Source NAT
- Правила MASQUERADE (динамический) и SNAT (фиксированный source IP)
- Поддержка алиасов в поле source (host / network / ipset / group)
- Идемпотентная проверка `iptables -C` предотвращает дублирование правил при перезапуске
- Авто-правила от туннельных интерфейсов отображаются как read-only бейджи `auto`

### Port Forwarding (DNAT)
- Запланировано — заглушка в UI

## Шлюзы (Gateways)

Мониторинг нескольких шлюзов и автоматическое переключение при отказе.

### Мониторинг шлюзов
- ICMP ping с настраиваемым интервалом и скользящим окном (потери пакетов %, задержка мс)
- HTTP/S проба — нативный Go `http.Get`, без subprocess curl
- Правила принятия решений о здоровье: `icmp_only`, `http_only`, `both_required`, `either`
- Live-статус в UI: online / degraded / offline + задержка + потери + HTTP-код

### Группы шлюзов
- Приоритет на основе уровней (tier 1 = основной, tier 2 = резервный, …)
- Типы триггеров: `packetloss`, `latency`, `packetloss_latency`
- Переключение на уровне группы: фоллбек активируется только когда ВСЕ участники уровня недоступны

### Автоматический фоллбек
- Флаг `fallbackToDefault` на каждом правиле файрвола
- При падении шлюза: вставка `blackhole default` или `default via <system-gw>` в PBR-таблицу
- 30-секундная задержка anti-flap перед восстановлением исходного маршрута шлюза
- Использует `ip route replace` (идемпотентно) — никогда не падает на устаревших маршрутах

## Файрвол

Единый список правил: фильтрация пакетов + Policy-Based Routing.

### Правила фильтрации
- Действия ACCEPT / DROP / REJECT
- Совпадение по: интерфейсу, протоколу (any/tcp/udp/icmp), источнику, назначению
- Источник / назначение: any, CIDR или алиас (host / network / ipset / group)
- L4 сопоставление портов через алиасы port / port-group
- Инвертированное совпадение (`not source`, `not destination`)
- Включение / выключение каждого правила
- Изменение порядка кнопками ↑ / ↓

### Policy-Based Routing (PBR)
- Назначение шлюза (или группы шлюзов) любому правилу → трафик помечается и направляется через этот шлюз
- Автоматически: `iptables MARK` + `ip route table N` + `ip rule fwmark N lookup N`
- `fallbackToDefault` — корректная деградация при недоступности шлюза

### Реализация
- Кастомные цепочки iptables: `FIREWALL_FORWARD` (filter) и `FIREWALL_MANGLE` (mangle/PREROUTING)
- `FIREWALL_FORWARD` вставляется на позицию 1 в цепочку FORWARD (перед всеми правилами интерфейсов)
- `_rebuildChains()` — атомарный flush + повторное применение при любом изменении

## Алиасы файрвола

Переиспользуемые именованные объекты для правил файрвола и NAT.

| Тип | Описание |
|-----|----------|
| `host` | Одиночные IP-адреса |
| `network` | CIDR-диапазоны |
| `ipset` | Большие наборы префиксов в kernel ipset (миллионы записей) |
| `group` | Объединяет несколько host / network алиасов |
| `port` | L4-порты (`tcp:443`, `udp:53`, `any:80`, `tcp:8080-8090`) |
| `port-group` | Объединяет несколько port-алиасов |

### Генерация ipset
- Ручная загрузка (один CIDR на строку)
- Автогенерация из **RIPE NCC** и **ipdeny** по стране или ASN
- Асинхронная задача с поллингом статуса
- Агрегация CIDR (collapse_addresses) перед загрузкой в ядро
- Снапшоты сохраняются на диск (`*.save`) — восстанавливаются автоматически при перезапуске контейнера

## Безопасность

### Аутентификация
- **Мультипользовательские аккаунты** — у каждого пользователя свой username, пароль, TOTP
- **TOTP (2FA)** — настройка через QR-код (Google Authenticator, Authy и др.)
  - Двухшаговый логин: пароль → 6-значный код
  - Включение / выключение на каждого пользователя, для отключения требуется текущий TOTP-код
- **API-токены** — долгоживущие bearer-токены для программного доступа
  - Формат: `ws_` + 64 hex-символа (256 бит энтропии)
  - В БД хранится только SHA-256 хеш — raw-значение показывается один раз
  - Временная метка `last_used` обновляется при каждом аутентифицированном запросе
  - Обходят TOTP — созданы для скриптов и автоматизации
  - Мгновенный отзыв из UI

### Сетевая безопасность
- `BIND_ADDR=127.0.0.1` — Cascade слушает только на localhost, не доступен напрямую
- **Обратный прокси Caddy** со скрытым префиксом `ADMIN_PATH`
  - Запросы к `/<secret>/...` → Cascade
  - Всё остальное → сайт-приманка (StreamVault)
- Ограничение скорости: 5 POST-запросов в минуту на IP (плагин caddy-ratelimit)
- TLS через **acme.sh** — сертификаты Let's Encrypt для bare IP-адресов
  - Профиль `shortlived`: срок действия 6 дней, автообновление каждые 3 дня
- `Referrer-Policy: no-referrer` — путь к админке не утекает на внешние сайты
- Контейнер Caddy: файловая система только для чтения, `cap_drop ALL`, только `cap_add NET_BIND_SERVICE`
- Session cookie: `SameSite=Strict` + `HttpOnly` — защита от CSRF и XSS

### Валидация входных данных
- Публичные ключи и PSK WireGuard валидируются как 32-байтные base64-строки перед передачей в shell
- CIDR-адреса (AllowedIPs) парсятся через `net.ParseCIDR` — отклоняются некорректные значения
- Endpoint валидируется на отсутствие shell-метасимволов (`;`, `|`, `&`, `` ` ``, `$`, и др.)
- ID интерфейсов автогенерируются системой (`wg10`, `wg11`, …) — пользователь не может задать произвольный ID

### Открытый режим
- Если в БД нет пользователей, все запросы проходят без аутентификации (удобство первого запуска)
- Как только создаётся первый пользователь, аутентификация применяется немедленно

## Администрирование (Admin Tunnel)

Легаси-интерфейс wg0 для традиционных admin VPN-клиентов.

- Управление классическими WireGuard клиентами (мобильные, ноутбуки)
- Генерация QR-кодов
- Включение / выключение / удаление клиентов

> **Примечание:** Бэкенд admin-туннеля частично мигрирован. Эндпоинт списка клиентов возвращает пустой массив в Go-переписи; полная миграция запланирована.

## Настройки

- Глобальные настройки: DNS, keepalive по умолчанию, AllowedIPs клиентов по умолчанию
- Пороги мониторинга шлюзов (глобальные значения по умолчанию)
- AWG2 Шаблоны: CRUD + установка дефолтного
- Генератор параметров AWG2 (⚡): 7 CPS-профилей, 3 уровня интенсивности, опциональное сохранение

## API

Полный REST API — каждое действие UI доступно программно.

- Аутентификация по сессии (cookie) для Web UI
- Аутентификация по Bearer-токену (`Authorization: Bearer ws_...`) для скриптов
- Все list-эндпоинты возвращают именованные обёртки (никогда не голый массив)
- Toggle через `PATCH { enabled: bool }` — минимальная нагрузка
- Ошибки возвращают `{ error: "message" }` с соответствующим HTTP-статусом

Полный справочник эндпоинтов: [API.md](API.md)

## Инфраструктура

### Runtime
- **Go 1.22** + **Fiber v2** — единый статический бинарник, без Node.js, без npm
- **SQLite** (modernc.org/sqlite, чистый Go, без CGO) — единый файл `wireguard.db`
- WAL journal mode — параллельное чтение, сериализованная запись
- Версионные миграции — схема безопасно эволюционирует между обновлениями
- `--network host` — UDP-порты WireGuard доступны без маппинга портов

### Контейнер
- Базовый образ: Alpine Linux
- Модули ядра AmneziaWG + WireGuard (DKMS, ядро хоста)
- Включены iproute2, iptables-nft, ipset
- Корректное завершение по SIGTERM / SIGINT

### Деплой
- `docker compose -f docker-compose.go.yml up -d`
- `BIND_ADDR=127.0.0.1` для деплоя за обратным прокси
- Директория данных монтируется в `/etc/wireguard/data` — переживает пересоздание контейнера
